# Session 6 Diagnosis — Agent Sales Pipeline blank cells

1. **API route feeding both surfaces is `/api/agent/pipeline-data`** (`app/api/agent/pipeline-data/route.ts`). The desktop page fetches it directly on mount; the mobile page receives the same payload via `useAgent()` → `AgentContext` → the same endpoint. The older `/api/agent/pipeline` route still exists but is gated on `requireRole(['developer','admin','super_admin'])` so agents never hit it.
2. **The API does return joined data.** It selects from `unit_sales_pipeline` with embedded `units` + `developments` joins, then mirrors the rows into a flat object. Fields present on the response: `salePrice`, `saleAgreedDate`, `depositDate`, `contractsIssuedDate`, `signedContractsDate`, `counterSignedDate`, `kitchenDate`, `snagDate`, `estimatedCloseDate`, `handoverDate`, `mortgageExpiryDate`. **Missing: `release_date` and `drawdown_date`** — they are never selected from the DB, never populated on the response.
3. **Sample shape** (flat, camelCased): `{ id, unitId, unitNumber, developmentId, developmentName, bedrooms, status, purchaserName, salePrice: 485000, saleAgreedDate: "2025-11-11", contractsIssuedDate: "2025-11-11", signedContractsDate: "2026-01-29", counterSignedDate: "2026-01-29", estimatedCloseDate: "2026-02-01", ... }`.
4. **Root cause is a shape mismatch on the desktop page.** `app/agent/dashboard/pipeline/page.tsx` declares `PipelineItem` with *nested* `prices.sale` and `dates.{saleAgreed, deposit, contractsIssued, contractsSigned, counterSigned, drawdown, handover, estimatedClose}`. Every column cell reads `p.prices?.sale` / `p.dates?.saleAgreed` etc. — but the API returns **flat** `salePrice` / `saleAgreedDate` / `signedContractsDate`. Every read therefore yields `undefined` → the formatters emit the em-dash. `getProgress()` counts zero milestones → always 0%. Revenue / Avg Price sum `0` across 86 rows → display as em-dash via `fmtCompact(0)`.
5. **Status normalisation also drops rows.** `normalizeStatus()` only keeps `for_sale / sale_agreed / contracts_issued / signed / sold` and `agreed` → `sale_agreed`. Real production values include `in_progress` (the "contracts issued but not signed" stage) and `handed_over`. Both fall through to `for_sale`, so the "In Progress" and "Complete" tiles under-count, and the Contracts / Complete filter tabs show nothing.
6. **No accidental filter drops rows.** The DB fetch has no `WHERE signed=...` gate; all 86 rows are returned. The bug is purely a key-path mismatch on the client plus two missing columns on the server. Fix: add `release_date` + `drawdown_date` to the API select, extend `normalizeStatus` to cover `in_progress` and `handed_over`, update `PipelineItem` + every cell reader on the desktop page to flat camelCase keys, and add real Revenue / Avg Price totals off the flat `salePrice`.
